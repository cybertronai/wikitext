DiffusionBlocks AR — results so far

Spec 17 replication on WikiText-103 byte-LM benchmark (300 s wall-clock, A100-80GB, val char-acc floor 0.70). Snapshot 2026-05-28.

Status: all four runs DQ — none cleared the 0.70 floor. Best val char-acc so far is 0.236 (first run) on 60k val chars. Per-byte argmax is hovering in the 0.22–0.25 band — well below the floor, but well above random (1/256 ≈ 0.004), so the model is learning something from the EDM + auxiliary-CE signal, just not enough at byte level under the 300 s cap.

Equi-probability partitioning is doing its job — block update counts are balanced (2.13–2.18k each). Block 0 converges as expected; block 2 (high noise) stays an order of magnitude higher and is the bottleneck for greedy-argmax accuracy, since inference's first Euler step starts at σ_max and depends on b2. This matches the spec's "high-σ block fails ⇒ paradigm doesn't transfer" failure mode.

What changed across versions

  • v1 → v3: trained longer/cleaner; val acc dropped slightly (0.236 → 0.221). No qualitative change.
  • v3 → v4: shortened σ_max from 80 → 5.0 (block boundaries [0.002, 0.178, 0.495, 5.0]). Training loss curves cleaner, eval accuracy roughly flat (~0.25 early-eval).
  • inference_fix: kept σ_max=80, attempted an inference-path fix; eval throughput collapsed from ~108 char/s to ~10 char/s — the fix made decode an order of magnitude slower and the eval did not complete inside the harness window. Investigate before any further variants.

Reading against spec 17's go/no-go criteria

Spec 17 calls for one remediation (B=2) on first DQ, then a DQ writeup with the failure mode. We are past the one-retry budget. The observed failure mode is the spec's case (1)+(2): the high-σ block does not denoise to anything the auxiliary CE head can argmax into correct bytes, and tightening σ_max (v4) didn't move the needle. Next steps per spec are not more variants — they're closing out with a DQ row and the per-block loss curves above as the record.
